FIX — Casos con id texto (caso_timestamp) no se eliminaban de Supabase
------------------------------------------------------------------------

DIAGNÓSTICO: La tabla casos tiene columna id tipo TEXT (no UUID). Supabase
acepta ids como 'caso_1781389524160' y los almacena correctamente. El guard
_isUUID introducido en v76/v77 bloqueaba el DELETE antes de llegar a Supabase
para cualquier id no-UUID, dejando esos records huérfanos en la nube.

Confirmado en Supabase Table Editor: 2 records con ids tipo caso_timestamp
y user_id UUID coincidente con el usuario → RLS pasaría sin problema, pero
el fetch nunca se ejecutaba.

FIX — _sbDeleteCasoCore: eliminar early return por id no-UUID (index.html):
  El bloque que detectaba ids no-UUID hacía return tras limpiar pending_deletes,
  sin ejecutar el fetch DELETE a Supabase. Removido el early return completo.
  La variable _isUUID se conserva solo para logging informativo.
  Un DELETE a un id inexistente retorna HTTP 200 con 0 filas — sin error.
  La limpieza de pending_deletes ocurre en el path exitoso normal.
