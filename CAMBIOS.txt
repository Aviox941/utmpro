FIX 1 — PERSISTENCIA DE SESIÓN: IndexedDB como respaldo del token de auth
--------------------------------------------------------------------------
PROBLEMA: Al cerrar y reabrir la PWA en Android, localStorage aparece vacío
(pension_utm_auth = AUSENTE). El SDK de Supabase no intenta recuperar la sesión
desde el servidor si no encuentra el token localmente → INITIAL_SESSION user=none
→ la app arranca sin sesión aunque el usuario ya había iniciado.

CAUSA: Chrome/Android puede limpiar localStorage de PWAs bajo presión de memoria.
IndexedDB tiene mayor prioridad de persistencia en este entorno.

IMPLEMENTACIÓN:
a) Bloque IDB restore (antes del SDK de Supabase, en <head>):
   - Al arrancar, si localStorage['pension_utm_auth'] está vacío,
     lee IndexedDB y restaura el token ANTES de que Supabase inicialice.
   - Supabase encuentra el token y restaura la sesión normalmente.

b) Custom storage adapter (_sbStorage) en createClient:
   - Reemplaza storage: window.localStorage por un objeto proxy.
   - Cada setItem('pension_utm_auth', ...) replica el valor a IDB (fire-and-forget).
   - Cada removeItem('pension_utm_auth') elimina de IDB también (logout limpio).
   - getItem sigue leyendo localStorage (síncrono, compatible con SDK).

LOGS ESPERADOS en debug panel:
   - Primer arranque post-login: "IDB: backup auth escrito — len=XXXX"
   - Arranque tras cierre con IDB poblado: "IDB: OK auth restaurado desde IDB"
   - Arranque sin sesión previa: "IDB: sin backup previo — arranque sin sesion"

FIX 2 — LOGS DUPLICADOS en debug panel
--------------------------------------------------------------------------
PROBLEMA: Cada mensaje del debug panel aparecía dos veces al arrancar.
CAUSA: dbg() real llamaba window._earlyDbgAdd(msg) cuando el panel ya existía,
pero el IIFE ya había volcado el _dbgBuffer al panel en DOMContentLoaded.
FIX: Eliminada la llamada a _earlyDbgAdd en dbg() real. Escribe directo al panel.
